# OneLap 批量导出 FIT 同步 Strava 设计文档

## 1. 背景与目标

用户需要一个本地一键运行的同步工具，从 OneLap 平台批量导出 FIT 文件并同步到 Strava。

首版目标：

- 一条命令运行：`python run_sync.py`
- 默认只处理最近 3 天活动，降低 OneLap 风控风险
- 增量同步，避免重复上传
- Strava 使用 OAuth 一次授权，后续自动刷新 token

## 2. 范围定义

### 2.1 首版包含

- 本地 Python CLI 一键执行
- OneLap 登录并拉取活动、下载 FIT
- 基于 `file_hash + start_time` 去重
- 上传 Strava（官方 REST API）
- 本地状态持久化与日志记录
- `--since YYYY-MM-DD` 手动覆盖时间窗口

### 2.2 首版不包含

- 云端定时任务部署
- GUI 界面
- 多平台账户聚合

## 3. 架构设计

采用分层模块化，核心同步流程与入口解耦，便于未来迁移到服务器定时任务。

- `run_sync.py`：CLI 入口，加载配置，触发同步，输出结果
- `onelap_client`：负责 OneLap 登录、活动查询、FIT 下载
- `dedupe_service`：计算文件哈希并生成活动指纹，执行去重
- `strava_client`：OAuth 授权与刷新、上传 FIT、查询上传状态
- `state_store`：持久化已同步记录、失败记录与同步元数据
- `sync_engine`：编排端到端流程并汇总执行结果

## 4. 数据流设计

单次执行流程：

1. 读取 `.env` 配置与本地状态
2. 计算本次拉取窗口
   - 默认窗口：最近 3 天
   - 实际 `since_date = max(用户配置/命令行 since, 最近成功同步时间)`
3. OneLap 拉取 `since_date` 之后活动并下载 FIT
4. `dedupe_service` 计算指纹并过滤已同步活动
5. `strava_client` 上传 FIT 到 Strava
6. 对上传结果轮询并分类（成功/失败）
7. 仅对成功上传活动写入状态
8. 输出统计（拉取数、去重后数、成功数、失败数）

## 5. 配置与安全

建议使用 `.env`：

- OneLap：账号、密码
- Strava：`CLIENT_ID`、`CLIENT_SECRET`、`REFRESH_TOKEN`、回调端口
- 同步参数：`DEFAULT_LOOKBACK_DAYS=3`、`MAX_ITEMS_PER_RUN`、请求间隔

安全要求：

- token 与敏感配置仅存本地，不写入代码仓库
- 日志中对敏感字段脱敏

## 6. 错误处理与重试策略

### 6.1 OneLap 侧

- 请求节流（固定间隔 + 随机抖动）
- 遇到风控/429/登录异常立即停止本轮，避免高频重试

### 6.2 Strava 侧

- 网络错误、5xx：指数退避重试（示例：10s/30s/90s）
- 4xx 参数类错误：不自动重试，直接失败并记录原因

### 6.3 一致性与幂等

- 仅在 Strava 确认成功后写入“已同步状态”
- 单条失败不阻塞全局任务
- 下次运行可继续处理未成功活动

## 7. 测试与验收标准

### 7.1 功能测试

- 首次 OAuth 成功并保存 token
- token 过期后自动刷新并继续上传
- 默认 3 天窗口生效
- `--since` 能覆盖默认窗口
- 重复执行不重复上传

### 7.2 异常测试

- OneLap 登录失败/风控响应时安全退出
- Strava 5xx 时按策略重试并记录
- 部分活动失败时输出失败清单

### 7.3 端到端验收

- 命令：`python run_sync.py`
- 输出示例：`拉取 8 条 -> 去重后 3 条 -> 成功 3 条 -> 失败 0 条`
- Strava 可看到新增活动；再次运行无重复数据

## 8. 后续演进（Phase 2）
